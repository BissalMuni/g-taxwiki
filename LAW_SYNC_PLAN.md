# 법령 개정 주기 확인·반영 자동화 — 설계서 (LAW_SYNC_PLAN)

> 상태: **초안(검토 대기)** · 작성일 2026-07-01
> 목적: 각 문서가 참조하는 법률·시행령·규칙의 **개정 여부를 주기적으로 법제처 API로 확인**하고, 개정 시 **해당 문서에 반영**하는 것을 자동·정기화한다.

## 1. 배경

- 본 프로젝트는 지방세 실무 매뉴얼(leaf TSX **207개**)이며, 각 문서는 특정 법령 조문을 근거로 한다.
- 법령은 수시 개정된다. 예) 2026.7.1 시행 지방세법 §13⑤③ 개정으로 고급주택 **시설기준(수영장·엘리베이터·에스컬레이터) 위임 근거 삭제** → luxury.tsx 수동 반영함(커밋 `e1f94cd`).
- 상위법 개정이 하위법령(시행령)보다 선행하는 **입법지연**도 존재하므로, 법률·시행령·규칙을 함께 추적해야 한다.

## 2. 확정된 방향 (2026-07-01 사용자 결정)

1. **개정 반영 방식**: 개정 감지 시 영향 문서에 **의견(코멘트) 자동 등록 → 기존 `review-feedback` 파이프라인**이 Claude로 반영. (기존 검증/커밋 인프라 재사용)
2. **참조 법령 소스**: **별도 레지스트리 JSON** (문서 ↔ 법령ID·조문·기준시행일 매핑).
3. **착수**: 계획서 먼저 → 승인 후 구현.

## 3. 아키텍처 개요

```
[cron] law-sync 워크플로
   │
   ├─(1) 레지스트리 JSON 로드 (문서별 참조 법령 목록)
   ├─(2) 법령별 현행 상태 조회 (법제처 lawSearch: 시행일자·공포번호·MST)
   ├─(3) 베이스라인과 비교 → 개정된 법령 검출
   ├─(4) 개정된 법령의 변경 조문 원문 조회 (법제처 lawService)
   ├─(5) 영향 문서마다 Supabase comments 에 "개정 반영 요청" 코멘트 자동 insert
   └─(6) 베이스라인 갱신(커밋)
                    │
                    ▼
[cron] 기존 review-feedback 워크플로 (그대로)
   → 미처리 코멘트 fetch → Claude 반영 → 검증 → 커밋 → processed 처리
```

## 4. 컴포넌트 상세

> **구조 결정: 모델 B** — 통합 파일엔 **법령만**(+기준 시행일), 문서↔법령 연결은 **각 문서 meta**에 둔다.

### 4.1 문서 meta의 참조 법령 (문서↔법령 연결)

각 leaf TSX의 `meta`에 자기가 근거하는 법령·조문을 구조화해 둔다.

```ts
export const meta = {
  // ...기존 필드...
  lawReference: [
    { law: "지방세법",       jo: "001300", label: "§13⑤③" },  // 제13조=001300
    { law: "지방세법 시행령", jo: "002800", label: "§28" },     // 제28조=002800
  ],
  lawLastAmended: "2026-07-01", // §4.6 헤더 표시용 (파이프라인이 갱신)
}
```
- `jo`: 법제처 조문 6자리.
- **문서↔법령 매핑은 이 meta가 단일 진실.** 별도 매핑 파일 없음.
- 부트스트랩: 기존 `meta.lawReference`(자유 텍스트)·본문 law.go.kr 링크를 스캔해 구조화하는 1회성 스크립트(`scripts/law-sync/normalize-meta.ts`) → 사람 검수.
- ⚠️ 확인: 현재 일부 문서는 meta가 `export const` 아니라 주석(`/** meta ... */`) 형태(예: luxury.tsx). 스캔·갱신이 되려면 meta 형식 통일 필요(별도 정규화 단계).

### 4.2 통합 법령 파일 (감시 목록 + 기준 시행일) — `src/lib/law-sync/law-baseline.json`

**법령만** 중복 없이 기록. 질문하신 "법령만 통합 기록된 파일".

```jsonc
{
  "지방세법":       { "lawId": "001649", "시행일자": "20260701", "공포번호": "제21308호", "mst": "282559" },
  "지방세법 시행령": { "lawId": "005077", "시행일자": "20260701", "공포번호": "제36445호", "mst": "287223" }
}
```
- 감시 대상 법령 집합 = **전 문서 meta.lawReference의 `law` 값을 모아 중복 제거**해 도출(스캔 시 자동). 이 파일은 그 각 법령의 **기준 시행일(베이스라인)**을 보관.
- 개정 감지 = (법제처 현행 시행일자·공포번호) ≠ (이 파일의 값). 최초 실행 시 현행값으로 시딩.
- 갱신은 코멘트 등록 성공 후 커밋(중복 등록 방지의 단일 기준).

### 4.3 개정 감지 스크립트 — `scripts/law-sync/check-amendments.ts`

**조사 범위는 딱 두 가지만: (1) 개정 여부, (2) 개정 이유.** (그 외 불필요한 조회는 하지 않음)

1. 전 문서 `meta.lawReference`를 스캔 → 감시 법령 집합 + (법령→문서) 역인덱스 구성.
2. 각 법령에 대해 법제처 `lawSearch.do`(현행) 호출 → `시행일자/공포번호/MST` 추출.
3. 베이스라인(`law-baseline.json`)과 비교 → **개정 여부** 판정, 변경된 법령 산출.
4. 변경된 법령만: **개정 이유 + 변경 조문**을 조회. 둘 다 **`lawService.do?target=law&MST=<mst>` 단일 응답**에 포함 — 별도 API 불필요.
   - 개정이유: 응답 `법령.제개정이유.제개정이유내용` (검증됨 — 2026 지방세법 개정이유에 "나. 고급주택 … **수영장 및 부대시설 요건을 삭제** … 대통령령으로 위임" 명시).
   - 변경 조문: 같은 응답 `법령.조문` 또는 `&JO=<jo>`로 해당 조문만.
   - 기본정보: `법령.기본정보`의 `공포번호/시행일자/제개정구분/공포일자`.
5. 변경 법령을 참조하는 문서를 역인덱스에서 수집 → §4.4로.
- ⚠️ **개정 감지는 `공포번호` 변화로 판정**할 것. `기본정보.시행일자`는 법령 전체 기준(예: 2026 지방세법=20260101)이라, 조문별 시차 시행(고급주택 §13⑤③=2026.7.1은 부칙에 존재)을 놓칠 수 있음. 공포번호가 바뀌면 새 개정으로 보고 개정이유를 확인하는 방식이 안전.
- 법제처 접근법·엔드포인트: 메모리 `beopjecheo-open-api` 참조 (OC=`minh0313`, 한글 percent-encoding, JSON 파일 저장 후 파싱).

### 4.4 코멘트 자동 등록

- Supabase `tax.comments`에 직접 insert(service role). 필드:
  - `content_path`, `feedback_type="content"`, `level="section"`, `flagged=false`
  - `author="법령개정봇"`
  - `section_title`: 근거 조문 라벨(예: `지방세법 §13⑤③ 개정 반영`)
  - `body`: **개정 이유(제개정이유)** + 변경 조문 원문 + "이 개정 내용을 문서에 반영하고 헤더 개정일을 갱신하라"는 지시 + 법제처 링크
- 개정 이유가 코멘트의 핵심 근거 — Claude가 "무엇이·왜 바뀌었는지"를 보고 정확히 반영하도록 함.
- 멱등성: 베이스라인 갱신 전까지만 등록되므로 같은 개정 재등록 없음. (스크립트 실패 시 베이스라인 미갱신 → 다음 회차 재시도)

### 4.6 문서 헤더에 "최근 개정 일시" 표시

각 문서 상단(헤드)에 그 문서가 근거하는 **법령의 가장 최근 개정/시행 일자**를 노출한다. 독자가 "이 문서는 언제 기준 법령인가"를 즉시 알 수 있게 한다.

- 값의 출처: 레지스트리의 `refs` 중 최신 `시행일자`(베이스라인에서 계산). 예) `지방세법 [시행 2026. 7. 1.] 기준`.
- 표시 방식(택1, §7-5 결정):
  - **(가) 콘텐츠 라인**: 각 leaf TSX 상단(h1 아래)에 소형 라인 삽입. 예:
    `<p className="text-sm text-muted">최근 법령 개정 반영: 2026. 7. 1. (지방세법 §13⑤ 일부개정)</p>`
    → 컴포넌트 변경 불필요. 개정 반영 코멘트에 "헤더 개정일 갱신" 지시를 포함해 파이프라인이 함께 갱신. *(권장 — 기존 인프라로 완결)*
  - **(나) meta 필드 + 공통 렌더**: `meta.lawLastAmended` 구조화 필드 추가 후 `ContentHeader`가 전 문서 일괄 렌더. 일관성↑이나 로더/컴포넌트 작업 필요(현재 일부 문서 meta가 주석 형태라 파싱 배선 확인 필요).
- 동기화: 개정 감지 시 본문 반영과 **같은 코멘트**로 헤더 일자도 갱신 → 항상 최신 유지.

### 4.5 GitHub Actions — `.github/workflows/law-sync.yml`

- `on: schedule`(예: 주 1회) + `workflow_dispatch`.
- 기존 `review-feedback.yml`보다 **앞선 시각**에 돌려, 등록된 코멘트가 곧이어 반영되게 배치.
- Secrets: `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, 법제처 `OC`.

## 5. 법제처 API 접근 제약 — OC 계정 승인 문제 (IP 제한 아님)

법제처 검증 실패의 원인은 **IP 화이트리스트가 아니라 OC 계정의 API 활용신청 승인 여부**였다.
- 실측(같은 IP `211.241.113.38`): OC `minh0313` → success, OC `gangubuytax` → "사용자 정보 검증 실패".
- 사용자 확인: **`minh0313`에는 IP 등록을 하지 않았음**에도 동작. 즉 `minh0313`은 승인된 계정이고 **IP 제한이 없다**. `gangubuytax`는 활용신청 미승인이라 실패(에러 메시지가 IP/도메인을 언급하지만 실제로는 계정 검증 실패의 일반 문구).

**함의**: IP 제한이 없으므로 **GitHub Actions(유동 IP)에서 `minh0313`으로 직접 호출 가능** → 고정 IP 머신 불필요, **전 과정 GH Actions 완전 자동화** 가능.

**남은 검증(P0)**: 로컬에서 GH Actions IP를 재현할 수 없으므로, 실제 GH Actions 러너에서 `minh0313` 호출이 성공하는지 **1회 스모크 테스트**로 확정한다. (혹시 실패하면 폴백: 고정 IP 머신 또는 고정 IP VPS 프록시.)

## 6. 단계별 구현 계획

- **P0. GH Actions 접근 검증**: 최소 워크플로로 GH 러너에서 `minh0313` 법제처 호출 성공 확인(§5). 성공 시 완전 자동화 확정, 실패 시 고정 IP 폴백.
- **P1. 레지스트리 부트스트랩**: `build-registry.ts`로 207개 문서의 `lawReference`/링크 스캔 → `law-registry.json` 초안 → 검수.
- **P2. 감지 코어**: `check-amendments.ts` (법제처 조회 + 베이스라인 비교 + 영향 문서 산출). 베이스라인 시딩.
- **P3. 코멘트 등록**: 변경 조문 조회 + Supabase insert. dry-run 모드 포함.
- **P4. 스케줄러**: 실행 환경 결정(§5) 후 cron 구성.
- **P5. 헤더 개정일 표시(§4.6)**: 표시 방식 확정 후, 코멘트 지시(가) 또는 공통 컴포넌트(나) 구현.
- **P6. 관측**: 등록 결과 로그/요약(어떤 법령·문서에 코멘트가 몇 건 등록됐는지).
